MCPs & Konnektoren — von einfach zu komplex
Zusatzmaterial zu Woche 3 — AI-Augmented PM


WAS IST MCP ÜBERHAUPT?

MCP steht für Model Context Protocol — ein offenes Protokoll das festlegt, wie Claude mit externen Diensten kommuniziert. Ohne MCP kann Claude nur das verarbeiten was du ihm direkt gibst. Mit MCP kann Claude aktiv in andere Systeme schauen: Slack lesen, Kalender abfragen, E-Mails suchen — ohne Copy-Paste.

Ein MCP-Server stellt Claude eine Liste von Tools zur Verfügung. Claude weiß welche Tools verfügbar sind und ruft sie auf. Die Tool-Namen folgen einem festen Muster:

    mcp__ServerName__tool_name

Beispiel: mcp__claude_ai_Slack__slack_read_channel


DIE STUFEN — VON EINFACH NACH KOMPLEX

Geh diese Liste von oben nach unten durch. Bleib so weit oben wie möglich. Je weiter unten, desto mehr Aufwand — und desto mehr kann schiefgehen (vor allem bei Authentifizierung).


Stufe 1: Konnektoren in der Desktop-App

Claude Desktop hat eine eingebaute Bibliothek von Konnektoren — fertig konfigurierte Verbindungen zu gängigen Diensten (Slack, Gmail, Google Calendar, Notion, …).

So erkennst du den Status:

"Konfigurieren"  →  Einmalig autorisieren → danach auch in Claude Code CLI verfügbar
"Verbunden"      →  Nur im Desktop verwendbar — in der CLI nicht verfügbar

Vorgehen:
1. Claude Desktop öffnen → Einstellungen → Connections
2. Dienst suchen und per Browser-Login autorisieren
3. Prüfen ob der Status auf "Konfigurieren" steht
4. In Claude Code CLI testen: "Welche MCP-Tools habe ich gerade?"

Wenn du mcp__claude_ai_Slack__* oder ähnliches siehst — es funktioniert.

Das ist der einfachste Weg. Immer zuerst probieren.


Stufe 2: MCP über /mcp add in der CLI konfigurieren

Wenn ein Konnektor "Verbunden" statt "Konfigurieren" zeigt, oder ein Dienst gar nicht in der Bibliothek ist, richtest du einen MCP-Server direkt in der CLI ein.

Befehl in Claude Code:     /mcp add
Befehl in der Kommandozeile:  claude mcp add

Beispiele:
    claude mcp add --transport http notion https://mcp.notion.com/mcp
    claude mcp add --transport stdio --env AIRTABLE_API_KEY=DEIN_KEY airtable -- npx -y airtable-mcp-server

Der MCP ist danach im Projektkontext oder global verfügbar — je nach Konfiguration.

Alternative: /plugin install
Wenn ein fertiges Plugin-Bundle für den Dienst existiert (MCP-Server + Skills + Hooks zusammen):
    /plugin install github
Weniger Konfiguration als /mcp add, aber auch weniger Kontrolle über was genau installiert wird.

Weiterführend: https://docs.anthropic.com/de/claude-code/mcp


Stufe 3: Custom MCP von Claude Code bauen lassen

Wenn kein fertiger Server existiert, kann Claude Code einen MCP-Server für dich schreiben — auf Basis der offiziellen MCP-Spezifikation.

So gehst du vor:
"Baue mir einen MCP-Server der [Dienst X] anbindet. Ich brauche folgende Tools: [was du tun willst]."

Claude Code erstellt die Serverstruktur, du richtest ihn lokal ein und registrierst ihn über /mcp add.

Aufwand: Mittel bis hoch. Sinnvoll wenn der Dienst eine API hat aber keinen Community-Server.


Stufe 4: Custom-Skript (Python, JavaScript, …)

Wenn MCP zu viel Overhead ist oder die Authentifizierung des Dienstes mit MCP-Servern schlecht harmoniert, lässt du Claude Code einfach ein Skript schreiben das direkt mit der API des Dienstes kommuniziert.

"Schreib mir ein Python-Skript das die letzten 10 Einträge aus [Dienst X] abruft und als JSON ausgibt."

Vorteil: Volle Kontrolle, kein MCP-Overhead, Fehler sind direkt sichtbar.
Nachteil: Kein nahtloser Claude-Zugriff — du rufst das Skript manuell auf oder baust eine Pipeline drumherum.


AUTHENTIFIZIERUNG — DER HÄUFIGSTE STOLPERSTEIN

Je weiter unten in der Liste, desto wahrscheinlicher wirst du auf Authentifizierungsprobleme stoßen:

- Token-Rotation — viele APIs erneuern Access-Tokens automatisch. Ein selbst gebauter Server oder ein Skript muss das explizit behandeln.
- OAuth-Flows — erfordern Browser-Redirects, die in CLI-Umgebungen nicht einfach funktionieren.
- Scope-Probleme — nicht alle Berechtigungen sind automatisch gesetzt; manche Dienste verlangen manuelle Freischaltung im Developer-Portal.

Faustregel: Wenn Stufe 1 oder 2 funktioniert, bleib da. Stufe 3 und 4 sind lösbar — aber plane Zeit ein.


ENTSCHEIDUNGSWEG AUF EINEN BLICK

Gibt es einen Konnektor in der Desktop-App?
  → Ja, Status "Konfigurieren"  → Stufe 1 — fertig.
  → Ja, Status "Verbunden"      → Stufe 2 (/mcp add) versuchen.
  → Nein ↓

Gibt es einen Community-MCP-Server für den Dienst?
  → Ja  → Stufe 2 (/mcp add)
  → Nein ↓

Hat der Dienst eine dokumentierte API?
  → Ja, MCP-Aufwand vertretbar  → Stufe 3 (Claude Code baut Server)
  → Ja, schnelle Lösung reicht  → Stufe 4 (Custom-Skript)
  → Nein → Kein programmatischer Zugriff möglich.


DER MERKSATZ

Konnektor → /mcp add → Custom Server → Custom Skript.
So weit oben bleiben wie möglich. Authentifizierung ist meistens das, was Zeit kostet — nicht die Logik.
